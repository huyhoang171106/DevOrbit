# Plan: Hoan thien DevOrbit theo README

## Muc tieu
Doi chieu cac hang muc README va trien khai phan con thieu/chua hoan chinh o muc production-ready hop ly: backend contract va data model, web admin/student UX va test, mobile registration/bookmark/ViewModel/DI nhe, ha tang va kiem thu.

## Phat hien hien trang
- Backend da co mot phan cac endpoint README noi thieu: `PublicTechStackController` ton tai nhung chi expose `/api/repos/{repoId}/tech-stacks`, chua co endpoint toan cuc `/api/tech-stacks`.
- Backend `AdminCourseController` dang co route base `/api/admin/courses` nhung thieu `GET` danh sach/detail cho admin.
- Backend `GithubRepo` va `TechStack` van la quan he one-to-many qua `tech_stacks.repo_id`; chua dung join table `repo_tech_stacks` dung nghia many-to-many.
- Backend `RepoCandidate` thieu metadata GitHub quan trong: description, primary language, topics, stars, forks, last pushed at, readme excerpt; `GithubScanService` chi luu owner/name/url/status.
- Backend admin repo edit API da co `PUT /api/admin/repos/{repoId}`, nhung web `AdminReposPage` chua co UI edit.
- Web co `HomePage` nhung route `/` van redirect `/courses`; routing con inline trong `App.tsx`.
- Web chua co test runner/test dependency.
- Web/mobile dang khai bao `techStacks: string[]`, trong khi backend tra `List<TechStackResponse>` object `{ name }`; day la mismatch co the lam UI hien thi `[object Object]` hoac mobile crash.
- Mobile thieu registration, bookmark, ViewModel state hoisting, DI framework; Gradle wrapper Unix `gradlew` chua xac minh nhung README noi thieu.
- Infrastructure dung default secret/password trong compose theo local-dev; can cai thien bang `.env.example`, validation/canh bao, va doc/test config thay vi hard-code production behavior.

## Nguyen tac thuc hien
- Lam backend API contract truoc, sau do cap nhat web va mobile theo contract moi.
- Uu tien thay doi nho nhung dung: khong lam knowledge graph/roadmap/notes/social vote neu README xep Phase 2+/tuong lai va khong co spec chi tiet.
- Khong pha du lieu local: voi schema many-to-many, can co chuyen doi/migration an toan neu dung Hibernate `update`; neu khong co Flyway, dung JPA mapping + data.sql idempotent va service fallback hop ly.
- Kiem thu bang test tu dong cho backend va web; mobile it nhat build/compile neu moi truong Gradle/Android cho phep.

## Ke hoach trien khai

### 1. Backend API va data model
1. Mo rong `RepoCandidate`:
   - Them columns: `description` TEXT, `primaryLanguage`, `topics` TEXT/JSON string, `stars`, `forks`, `lastPushedAt`, `readmeExcerpt` TEXT.
   - Cap nhat `RepoCandidateResponse` de tra cac field moi cho web admin.
   - Cap nhat `GithubScanService` doc GitHub Search API fields: `description`, `language`, `topics`, `stargazers_count`, `forks_count`, `pushed_at`.
   - Neu kha thi, goi them GitHub contents/readme endpoint cho tung candidate voi timeout ngan va fail-soft de lay excerpt; neu rate/error thi de null, khong fail ca scan.
   - Tang xu ly loi null-safe cho `items`, `full_name`, duplicate URL.
2. Chuyen repo-tech stack sang many-to-many:
   - Cap nhat `GithubRepo` co `Set<TechStack> techStacks` voi `@ManyToMany` va `@JoinTable(name = "repo_tech_stacks")`.
   - Cap nhat `TechStack` thanh danh muc stack co unique `name`, bo phu thuoc bat buoc vao repo; neu can backward compatible tam thoi, khong dung `repo_id` nua trong logic moi.
   - Cap nhat `TechStackRepository`: `findDistinctBy...`, `findByNameIgnoreCase`, query tech stacks by repo id qua join table, delete association khong delete danh muc stack.
   - Cap nhat `GithubRepoService`, `RepoCandidateService`, `CourseService` mapping response tu collection many-to-many.
   - Cap nhat query filter repo by tech stack dung join relationship.
3. Bo sung endpoint tech stack toan cuc:
   - `GET /api/tech-stacks` tra distinct tech stack active/known, sap xep theo ten.
   - Giu `/api/repos/{repoId}/tech-stacks` de tuong thich.
4. Bo sung admin course read endpoints:
   - `GET /api/admin/courses` tra danh sach course cho admin, bao gom active/inactive neu service ho tro; neu chua, it nhat tra active va dung endpoint rieng cho admin.
   - `GET /api/admin/courses/{id}` tra detail.
5. Hoan thien admin repo update:
   - Mo rong `ApprovedRepoUpdateRequest` neu can: `displayName`, `description`, `githubUrl`, `primaryLanguage`, `stars`, `active`, `techStacks`.
   - Validate input, trim tech stack, dedupe case-insensitive.
6. Backend tests:
   - Cap nhat test hien co theo DTO moi.
   - Them/bo sung tests cho `/api/tech-stacks`, admin course `GET`, scan candidate metadata mapping, approve candidate tao association many-to-many, update repo tech stacks.

### 2. Web frontend
1. Sua type contract:
   - Doi `RepoSummary.techStacks` va candidate metadata type de phu hop backend object hoac normalize trong `api.ts` ve `string[]` mot cach nhat quan.
   - Uu tien normalize tai API client/helper de UI hien co it bi sua.
2. Tach router:
   - Tao `src/router.tsx` chua route tree.
   - Cap nhat `App.tsx` chi render `BrowserRouter` + `Layout` + router.
   - Wire `/` vao `HomePage` thay vi redirect.
3. Hoan thien homepage:
   - Bien `HomePage` thanh landing page dung tinh than README: gioi thieu catalog course, legacy repos, admin review pipeline, CTA browse/login/bookmarks.
   - Dam bao responsive mobile/desktop va giu visual language hien co.
4. Admin repo edit UI:
   - Them modal/form edit repo trong `AdminReposPage`/component rieng.
   - Cho sua display name, description, primary language, stars, GitHub URL neu API cho phep, tech stacks dạng comma/chips, active.
   - Hien thi loi API, loading state, validate URL/required.
5. Candidate UI metadata:
   - Cap nhat `CandidateTable`/`ApproveModal` hien description, language, stars/forks, topics, pushed date, readme excerpt de admin review co du thong tin.
   - Prefill approve description/tech stacks tu candidate metadata neu co.
6. Test web:
   - Them Vitest + React Testing Library + jsdom vao devDependencies/script `test`.
   - Viet tests cho API error handling/normalization, router homepage route, admin repo edit submit, candidate metadata render neu pham vi thoi gian cho phep.

### 3. Mobile app
1. Sua deserialization tech stacks:
   - Tao model `TechStackResponse(name: String)` hoac custom adapter.
   - Doi `RepoSummary.techStacks` thanh list object hoac expose computed display names trong UI.
   - Cap nhat `RepoListSection`, `RepoFilterSheet`, `RepoDetailScreen` dung `.name`.
2. Them registration:
   - Model `StudentRegisterRequest`.
   - ApiService `POST /api/student/register`.
   - Repository method `registerStudent` luu session neu API tra token.
   - UI `RegisterScreen`, link tu login, validation mssv/email/password/fullName va loi API.
3. Them bookmark:
   - Model `BookmarkRequest`, `BookmarkResponse`, `BookmarkTargetType`.
   - ApiService: `GET /api/student/bookmarks`, `POST /api/student/bookmarks`, `DELETE /api/student/bookmarks/{id}` hoac theo API hien co sau khi doc chi tiet controller.
   - Repository methods co Bearer token tu `AuthSessionStore`.
   - UI bookmark/unbookmark cho course/repo detail, va `BookmarksScreen` danh sach co dieu huong.
4. ViewModel va state hoisting:
   - Tao ViewModels cho auth, course list/detail, bookmarks su dung `viewModel()` va `UiState` sealed/data classes.
   - Chuyen state/network call ra khoi composable chinh `DevOrbitApp`, `LoginScreen`, `CourseListScreen`.
5. DI framework:
   - Can nhac Hilt vi README yeu cau DI; them plugin/dependencies Hilt, `@HiltAndroidApp`, modules provide ApiService/Repository/Stores.
   - Neu Hilt lam tang rui ro build qua lon, toi thieu tao lightweight dependency container ViewModel factory; tuy nhien de dung README nen uu tien Hilt.
6. Gradle wrapper Unix:
   - Kiem tra `devorbit-mobile/gradlew`; neu thieu, tao/cap nhat wrapper script chuan tu Gradle wrapper hien co.
7. Mobile validation:
   - Chay Gradle build/test neu Android SDK san sang; neu khong, chay cac check co the va bao ro gioi han.

### 4. Infrastructure va cau hinh
1. Them `.env.example` o root voi `GITHUB_TOKEN`, `JWT_SECRET`, `DATABASE_*`, `POSTGRES_*` va ghi chu local/prod.
2. Cap nhat `docker-compose.yml` de dung env substitution cho DB username/password thay vi hard-code truc tiep, co default local dev ro rang.
3. Xem lai `application.yaml`: giu default local nhung can canh bao/validate secret ngan trong production profile neu co profile.
4. Test integration:
   - Neu backend tests dang can PostgreSQL localhost, xem kha nang dung Testcontainers hoac profile test voi H2 khong phu hop PostgreSQL. Uu tien cai thien test config de co the chay on-demand bang Docker Compose DB.

### 5. Verification
1. Backend: chay `mvn test` trong `devorbit-api`.
2. Web: chay `npm install` neu dependencies thay doi, `npm run build`, `npm test -- --run`.
3. Mobile: chay `gradlew.bat test`/`assembleDebug` neu moi truong Android SDK san sang.
4. Full stack: neu kha thi, `docker compose config` va `docker compose up -d --build` de xac minh service start; neu ton thoi gian/rui ro, it nhat build tung service.
5. Kiem tra git diff de dam bao khong chinh sua ngoai pham vi va khong commit.

## Rui ro va cach giam thieu
- Chuyen many-to-many co the anh huong schema existing; se giu ten table moi `repo_tech_stacks`, khong xoa bang/column cu trong code migration thu cong de tranh mat du lieu.
- Hilt tren mobile co the doi thay doi Gradle/plugin nhieu; se lam theo phien ban tuong thich Kotlin 2.0.21/AGP hien co va build verify.
- GitHub README excerpt co the cham/rate limit; se fail-soft va gioi han per scan.
- Web/mobile type mismatch can dong bo backend response; se chon normalization nhat quan va them test.
